INFORME TÉCNICO: EVALUACIÓN DE LA POSTURA DE SEGURIDAD

Proyecto: PAI-5 RedTeamPro
Cliente: Organización Pública de Andalucía
Auditores (Red Team): INSEGUS (Security Team 10)
Fecha: 21 de mayo de 2026

1. Resumen Ejecutivo

Este informe detalla el proceso y los resultados de la auditoría de seguridad del tipo caja negra (Blackbox) realizada a los sistemas de información de una organización pública de la Junta de Andalucía. La evaluación ha sido llevada a cabo por el equipo de seguridad INSEGUS (Red Team) siguiendo las directrices técnicas recogidas en la guía estándar NIST SP 800-115 (Technical Guide to Information Security Testing and Assessment).

El objetivo primordial del ejercicio ha sido identificar, analizar y explotar vulnerabilidades de manera realista para evaluar la resistencia de los sistemas frente a amenazas y determinar el impacto estratégico de un hipotético compromiso de la red interna de la organización.

Advertencia Legal
Este informe se redacta con fines puramente formativos y didácticos. Las pruebas de seguridad se han ejecutado sobre una infraestructura virtualizada local debidamente controlada y autorizada. El uso de estas técnicas sobre redes ajenas sin el debido consentimiento formal constituye un delito informático.

2. Fase 1: Planificación y Alcance

De acuerdo con el estándar NIST 800-115, toda evaluación técnica requiere definir formalmente el escenario de trabajo, las reglas de compromiso y el alcance del análisis.

2.1 Escenario de Trabajo y Reglas de Compromiso

  • Modalidad: Caja Negra (Blackbox Pentesting). El Red Team no dispone de credenciales previas, mapas de red o código fuente antes del inicio de la prueba.
  • Límites de Ejecución (Scope): Las pruebas se realizan de forma dirigida sobre el host loopback asignado por la organización (127.0.0.1).
  • Exclusión de Daños: Se prohíben las pruebas de denegación de servicio (DoS) agresivas que puedan interrumpir los servicios de administración pública.
  • Reproducibilidad: Toda prueba realizada debe registrar evidencias y logs auditables para que el personal del Blue Team pueda verificar la secuencia de eventos e implementar parches.

2.2 Entorno Tecnológico Target

Para evitar dependencias externas e imposición de licencias pesadas en el sistema de corrección, el Red Team ha desarrollado un entorno simulado interactivo en memoria que recrea con precisión los servicios y las respuestas de red de los siguientes demonios:

  • FTP: ProFTPD versión 1.3.5 en el puerto 2121.
  • HTTP: Servidor Web Apache v2.4.7 con script expuesto /debug.php en el puerto 8080.
  • Database: MySQL Server versión 5.5.47 en el puerto 3306.

3. Metodología Usada, Herramientas y TTPs

Para garantizar la validez metodológica y la reproducibilidad técnica de las pruebas de penetración, se empleó un ciclo estructurado y se mapearon las actividades al framework estándar MITRE ATT&CK.

3.1 Ciclo Metodológico NIST SP 800-115

  1. Planificación (Planning): Establecimiento del alcance, objetivos del Security Team y límites del sandbox.
  2. Descubrimiento (Discovery): Reconocimiento de puertos abiertos, obtención de banners perimetrales y fingerprinting del OS.
  3. Ejecución (Attack / Exploitation): Explotación del vector inicial de red, inyección de la backdoor, post-explotación y escalado de privilegios.
  4. Informes (Reporting): Desarrollo del análisis de impacto, plan de mitigación técnica y directrices de remediación para la organización pública.

  • Python standard library (socket, urllib): Empleado como suite de red de bajo nivel.

5. Fase 3: Análisis de Vulnerabilidades

Una vez obtenidos los banners y las versiones exactas, el Red Team contrastó esta información con bases de datos públicas de vulnerabilidades (CVE del MITRE y NVD de NIST).

Vulnerabilidad Crítica en FTP: CVE-2015-3306 (ProFTPD mod_copy)

Clasificación: Crítica (CVSS 10.0) | CWE: CWE-284

Descripción: El módulo mod_copy en ProFTPD 1.3.5 permite a usuarios no autenticados ejecutar comandos SITE CPFR (Copy From) y SITE CPTO (Copy To). Esto permite copiar cualquier archivo del sistema a cualquier otra ubicación visible por el demonio del servidor web, facilitando la creación de web shells.

Impacto: Compromiso del perimetro web y escritura no autorizada en directorios públicos.

Exposición de Diagnóstico: RCE en debug.php

Clasificación: Alta (CVSS 9.8) | CWE: CWE-94

Descripción: El script de diagnóstico interno expone un parámetro cmd que ejecuta comandos de sistema operativo directamente mediante funciones como system() sin desinfectar la entrada.

Impacto: Ejecución remota de comandos bajo el contexto de www-data de manera directa.

Exposición de Base de Datos MySQL Externa

Clasificación: Media (CVSS 5.0) | CWE: CWE-200

Descripción: El puerto de base de datos MySQL (3306) es accesible de forma directa a través de la red en interfaces externas, facilitando ataques de fuerza bruta o escaneo de configuraciones.

Impacto: Fugas de información e intentos persistentes de descifrado de credenciales.

6. Fase 4: Explotación y Escalada de Privilegios

6.1 Ejecución del Ataque (Cadena de Explotación)

El proceso de intrusión se automatizó mediante un exploit en Python que imita el comportamiento de los módulos de explotación de Metasploit (proftp_modcopy_exec):

  1. Copia del Backdoor vía FTP: Se copió el archivo de diagnóstico debug.php a la ubicación pública de Apache con el nombre de backdoor.php explotando el mod_copy de ProFTPD mediante los comandos SITE CPFR / SITE CPTO.
  2. Establecimiento de Command & Control (C2): Se interactuó con la web shell haciendo peticiones HTTP GET estructuradas a http://127.0.0.1:8080/backdoor.php?cmd=<comando>.
  3. Lectura de Datos y Escalada de Privilegios: Al ejecutar sudo -l, se observó que el usuario www-data poseía permisos para ejecutar /usr/bin/python3 como root sin contraseña. Se aprovechó esta vía para ejecutar un comando y leer el flag restringido en /root/flag_root.txt.

8. Plan de Acción y Mitigaciones Recomendadas (Blue Team)

Acciones Inmediatas Requeridas
Se recomienda encarecidamente desactivar el módulo mod_copy de FTP, deshabilitar la cuenta sudo interactiva de www-data y restringir las funciones peligrosas de ejecución de comandos de PHP.

8.1 Mitigación de FTP (ProFTPD mod_copy)

Comentar la carga de mod_copy en /etc/proftpd/modules.conf:

# LoadModule mod_copy.c

O restringir los comandos en proftpd.conf:

<Limit SITE_COPY>
    DenyAll
</Limit>

8.2 Deshabilitar ejecución de comandos en PHP

En el archivo de configuración php.ini, establecer la directiva:

disable_functions = exec,passthru,shell_exec,system,proc_open,popen,curl_exec,curl_multi_exec,parse_ini_file,show_source

8.3 Corrección en directiva de sudoers

Eliminar la línea que da acceso ilimitado de python3 como root a www-data en /etc/sudoers para evitar la escalada de privilegios directa.

8.4 Hardening de Red en MySQL

Restringir el acceso externo al puerto 3306 configurando la directiva de escucha en local en my.cnf:

bind-address = 127.0.0.1

9. Uso Estratégico de los Resultados de la Auditoría

Los resultados obtenidos a través de las pruebas de penetración deben utilizarse por la organización pública en las siguientes áreas de gobernanza de seguridad:

  • Como punto de referencia para la adopción de medidas correctivas: Proporciona un listado ordenado y verificado empíricamente de debilidades de seguridad urgentes que requieren atención de la administración de sistemas.
  • En la definición de las actividades de mitigación para hacer frente a las vulnerabilidades identificadas: Guía en la creación de políticas de hardening efectivas y la planificación de actualizaciones de servicios desactualizados.
  • Como punto de referencia para el seguimiento del progreso de una organización en el cumplimiento de los requisitos de seguridad: Establece un registro histórico comparable ("baseline") para valorar el avance en auditorías futuras.
  • Para evaluar el estado de aplicación de los requisitos de seguridad del sistema: Permite validar técnicamente de manera independiente si las reglas lógicas (por ejemplo, menor privilegio) están operativas en producción.
  • Para realizar análisis de costes y beneficios de las mejoras de la seguridad del sistema: Aporta evidencias objetivas a la directiva para justificar el gasto en herramientas defensivas en comparación al alto coste estratégico de un compromiso total.
